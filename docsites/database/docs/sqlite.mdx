# SQLite — Pokemon Data

## Overview

The SQLite database (`pokemon-data.sqlite3.db`) contains the complete Pokemon TCG card catalog. It is populated offline by the `apps/scripts` data sync process and mounted read-only into the `rest-api` and `graphql-api` containers at runtime.

## Tables

### `pokemon_cards`

Stores individual card records. Key columns:

| Column | Type | Description |
|--------|------|-------------|
| `id` | TEXT | Canonical card ID (e.g., `sv3-123`) |
| `name` | TEXT | Card name |
| `supertype` | TEXT | `Pokémon`, `Trainer`, or `Energy` |
| `subtypes` | TEXT | JSON array |
| `hp` | INTEGER | Hit points (Pokémon only) |
| `types` | TEXT | JSON array of energy types |
| `set_id` | TEXT | Foreign key to `pokemon_card_sets.id` |
| `rarity` | TEXT | Rarity string |
| `number` | TEXT | Collector number within the set |
| `attacks` | TEXT | JSON array |
| `abilities` | TEXT | JSON array |

### `pokemon_card_sets`

Stores set (expansion) records. Key columns:

| Column | Type | Description |
|--------|------|-------------|
| `id` | TEXT | Set ID (e.g., `sv3`) |
| `name` | TEXT | Set name |
| `series` | TEXT | Series name (e.g., `Scarlet & Violet`) |
| `printed_total` | INTEGER | Card count printed on packaging |
| `total` | INTEGER | Total cards including secret rares |
| `release_date` | TEXT | `YYYY/MM/DD` |
| `ptcgo_code` | TEXT | PTCGO set code |

## Access Pattern

Both `rest-api` and `graphql-api` open the database with `bun:sqlite` in read-only mode. Writes at runtime are not possible — `DATABASE_READONLY=true` is enforced in the environment.

## Updating Card Data

The database file is mounted as a volume, not baked into any image. Updating it does not require rebuilding Docker images.

For the full step-by-step process including journal mode reset and production deployment, see the [Rolling Out a New Card Set](/deployment/rollout-sqlite) runbook.

Summary:

1. Run `bun run docker:scripts:db:sync` to produce a new `pokemon-data.sqlite3.db`
2. Reset journal mode to `DELETE` (required — containers open the file read-only from a directory they cannot write to)
3. Copy to `/opt/deckvault/database/` with `deckvault:deckvault` ownership and `644` permissions
4. `docker compose restart rest-api graphql-api`

<Callout variant="warning">
  Never deploy a SQLite file that was last opened in WAL journal mode. The containers will fail with `SQLITE_READONLY_DIRECTORY` because WAL mode requires write access to the directory to create `-shm` sidecar files, which the `bun` process (uid=1000) cannot do in a directory owned by `deckvault` (uid=996).
</Callout>
